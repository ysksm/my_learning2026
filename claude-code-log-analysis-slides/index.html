Claude Code は全セッションを JSONL で記録しています。その在り処・スキーマ・分析手順を step by step で辿り、会話・ツール・トークン・時間・エラーまで「解析できるもの全部」を取り出します。

jq + Python だけでOK
このスライドのゴールは、
自分のログから利用実態レポートを自力で作れること。
よく使うツール、かかったコスト、待ち時間、失敗傾向まで数字で見えるようになります。
CC-LOG / 00START

01 / まず在り処を知る

ログはどこにある？ — ~/.claude/ の地図

主役はプロジェクトごとのセッショントランスクリプト。作業ディレクトリのパスを /→- に変換したフォルダに、セッションIDごとの JSONL が置かれます。

~/.claude/ ├── projects/ ← ★ 本命：セッショントランスクリプト │ └── -home-user-my_learning2026/ (cwd の / を - に置換した名前) │ └── <session-id>.jsonl (1セッション = 1ファイル) ├── history.jsonl ← 入力プロンプトの履歴 ├── todos/ ← TodoWrite のタスク状態 (JSON) ├── shell-snapshots/ ← Bash 実行時のシェル環境スナップショット └── settings.json ← cleanupPeriodDays で保持期間を制御
注意：古いトランスクリプトは cleanupPeriodDays（既定30日）で自動削除されます。長期分析したい場合は先にコピーして退避しましょう。

02 / STEP 1

JSONL の基本 — 1行 = 1レコード

JSONL は「1行に1つの JSON オブジェクト」を並べた形式。行単位で読めるので、jq やストリーム処理と相性抜群です。まずはセッション一覧と行数から。

# プロジェクトのログディレクトリへ
cd ~/.claude/projects/-home-user-my_learning2026

# セッション一覧（更新順）とレコード数
ls -lt *.jsonl
wc -l *.jsonl

# 最初の1レコードを整形して眺める
head -1 <session-id>.jsonl | jq .

ここで分かること

  • セッションの数と規模（行数≒イベント数）
  • ファイル更新日時＝最後に使った日
  • プロジェクトごとの利用量の偏り

準備するツール

jq（集計ワンライナー用）と Python 3（本格分析用、pandas があると便利）。どちらも標準的な環境で入手できます。

レコード種別を把握する — type フィールド

全レコードはトップレベルの type で分類できます。まず種別の分布を数えると、そのセッションの構造が一目で分かります。

jq -r '.type' *.jsonl | sort | uniq -c | sort -rn
type意味中身のポイント
userユーザー発話 or ツール実行結果message.content が文字列なら人間の入力、配列（tool_result）ならツールの戻り
assistantClaude の応答message.content[] に text / thinking / tool_use ブロック。usage（トークン）と model 付き
attachmentコンテキスト添付システムリマインダーや差し込み情報
queue-operation入力キュー操作enqueue/dequeue とプロンプト原文
summary / last-prompt などセッション管理用コンパクト時の要約、再開用ポインタ

05 / STEP 4

会話ツリーを復元する — parentUuid

各レコードは parentUuid で直前のレコードを指す連結リスト（分岐すると木）です。これを辿ると会話の実際の流れが再現できます。

user
prompt (parent=null)
→
assistant
thinking / text
→
assistant
tool_use
→
user
tool_result
→
assistant
text（最終回答）
import json
nodes = {}
with open(path) as f:
    for line in f:
        d = json.loads(line)
        if 'uuid' in d: nodes[d['uuid']] = d

def chain(leaf):  # 葉から根まで遡る
    while leaf: yield leaf; leaf = nodes.get(leaf.get('parentUuid') or '')
sidechain：isSidechain: true の系列はサブエージェント（Task/Agent）の独立した会話。本流と分けて集計すると「委譲した仕事量」が測れます。
CC-LOG / 05STEP 4 — CONVERSATION TREE

06 / STEP 5

ユーザープロンプトを抽出する

「人間が何を頼んだか」はプロダクト分析の出発点。type=user のうち content が文字列のものが人間の入力、配列のものはツール結果なので除外します。

# 人間のプロンプトだけを時刻付きで抽出
jq -r 'select(.type=="user" and (.message.content|type)=="string")
       | [.timestamp, .message.content] | @tsv' *.jsonl

# 全プロジェクト横断ならこちら（入力履歴）
jq -r '.display' ~/.claude/history.jsonl | head -50

ここで分かること

  • 依頼の種類（実装 / 調査 / 修正 / 質問）の割合
  • 1セッションあたりの指示回数＝往復数
  • プロンプトの長さと成果の関係

ワンランク上

抽出したプロンプトを Claude に分類させる（「このプロンプト集を用途別にタグ付けして」）と、自分の使い方の癖が定量化できます。

CC-LOG / 06STEP 5 — PROMPTS

07 / STEP 6

ツール使用を分析する — tool_use と tool_result

assistant の tool_use ブロック（name / input / id）と、それに対応する user 側の tool_result（同じ tool_use_id）をペアにすると、Claude の「行動ログ」になります。

# ツール別の使用回数ランキング
jq -r 'select(.type=="assistant") | .message.content[]?
       | select(.type=="tool_use") | .name' *.jsonl | sort | uniq -c | sort -rn

# Bash で実行されたコマンドの傾向（先頭単語）
jq -r 'select(.type=="assistant") | .message.content[]?
       | select(.type=="tool_use" and .name=="Bash")
       | .input.command' *.jsonl | awk '{print $1}' | sort | uniq -c | sort -rn

行動の内訳

Read / Edit / Write / Bash / Grep… の比率で「読む仕事か書く仕事か」が見える。

触ったファイル

input.file_path を集計すればホットスポットのファイルが特定できる。

実行結果

toolUseResult の stdout / stderr / interrupted で成否と中断を判定。

08 / STEP 7

トークンとコストを算出する — usage

assistant レコードの message.usage に、APIと同じ粒度のトークン内訳が毎回記録されています。model と掛け合わせれば概算コストが出せます。

"usage": {
  "input_tokens": 2,
  "output_tokens": 410,
  "cache_creation_input_tokens": 39732,
  "cache_read_input_tokens": 0,
  "cache_creation": {
    "ephemeral_1h_input_tokens": 39732,
    "ephemeral_5m_input_tokens": 0 },
  "service_tier": "standard"
}
# セッション合計（入力/出力/キャッシュ）
jq -s '[.[] | select(.type=="assistant")
        | .message.usage]
  | { in:  (map(.input_tokens)|add),
      out: (map(.output_tokens)|add),
      cache_w: (map(.cache_creation_input_tokens)|add),
      cache_r: (map(.cache_read_input_tokens)|add) }' \
  <session-id>.jsonl
  • モデル別（.message.model）に分ければ単価を掛けてコスト概算
  • cache_read ÷ (input+cache_read) でキャッシュヒット率＝節約効率
  • ブランチ・プロジェクト別に集計すれば「この機能開発に幾らかかったか」
近道：コスト集計だけなら OSS の npx ccusage が同じ JSONL を読んで日別・モデル別レポートを出してくれます。

09 / STEP 8

時間を分析する — timestamp の差分

全レコードにミリ秒精度のタイムスタンプがあるので、差分を取るだけで応答速度・作業時間・活動パターンが数値化できます。

応答レイテンシ

ユーザープロンプト → 直後の assistant レコードの時刻差。ターンごとの「待ち時間」。

ツール実行時間

tool_use → 対応する tool_result の時刻差。重いビルドやテストが特定できる。

セッション時間

最初と最後のレコードの差。日別・時間帯別に並べれば利用ヒートマップに。

import json, pandas as pd
rows = [json.loads(l) for l in open(path)]
df = pd.DataFrame([r for r in rows if 'timestamp' in r])
df['ts'] = pd.to_datetime(df['timestamp'])
df['gap'] = df['ts'].diff()          # レコード間隔
df.groupby(df['ts'].dt.hour).size()   # 時間帯別アクティビティ

10 / STEP 9

エラーと中断を洗い出す

失敗パターンの分析は改善効果が最も大きい部分。ログには失敗のシグナルが複数の場所に残ります。

シグナル場所意味
is_error: trueuser レコードの tool_result ブロックツール呼び出しがエラーを返した
stderr が非空toolUseResultコマンドは動いたが警告/エラー出力あり
interrupted: truetoolUseResultユーザーが実行を中断した（信頼のシグナル）
stop_reasonassistant の messagetool_use / end_turn など応答の終わり方
権限拒否の文言tool_result の本文許可プロンプトで拒否された操作の傾向
# エラーになったツール呼び出しの数
jq '[select(.type=="user") | .message.content[]?
     | select(.type=="tool_result" and .is_error==true)] | length' *.jsonl
使い方：「エラー率の高いツール×コマンド」を出すと、CLAUDE.md に書くべき注意書き（例：このリポジトリではテストはこう実行する）が自動的に決まります。

11 / STEP 10

可視化して1枚のレポートにする

ここまでの集計を pandas に集めれば、あとは好きな形で出力するだけ。まずは4枚のチャートから始めるのがおすすめです。

最初に作る4チャート

  • 日別トークン消費（積み上げ：入力/出力/キャッシュ）
  • ツール使用回数 Top10（横棒）
  • 時間帯×曜日の利用ヒートマップ
  • セッションごとの往復数とエラー数（散布図）
import glob, json, pandas as pd
recs = []
for p in glob.glob('~/.claude/projects/*/*.jsonl'):
    for l in open(p):
        d = json.loads(l)
        if d.get('type')=='assistant':
            u = d['message'].get('usage',{})
            recs.append({'ts':d['timestamp'],
              'model':d['message'].get('model'),
              'out':u.get('output_tokens',0)})
df = pd.DataFrame(recs)  # → groupby して plot
発展：この解析自体を Claude Code に頼むのが最速です。「~/.claude/projects の JSONL を読んで利用レポートを HTML で作って」— このスライドの手順がそのままプロンプトになります。

13 / チートシート

コピペで使えるワンライナー集

# ① レコード種別の分布
jq -r '.type' s.jsonl | sort | uniq -c

# ② 人間のプロンプト一覧
jq -r 'select(.type=="user" and
  (.message.content|type)=="string")
  | .message.content' s.jsonl

# ③ ツール使用 Top10
jq -r 'select(.type=="assistant")
  | .message.content[]?
  | select(.type=="tool_use").name' s.jsonl \
  | sort | uniq -c | sort -rn | head
# ④ 総出力トークン
jq -s '[.[] | select(.type=="assistant")
  | .message.usage.output_tokens] | add' s.jsonl

# ⑤ 使用モデルの内訳
jq -r 'select(.type=="assistant")
  | .message.model' s.jsonl | sort | uniq -c

# ⑥ エラーになった tool_result 数
jq '[select(.type=="user")
  | .message.content[]?
  | select(.type=="tool_result"
     and .is_error==true)] | length' s.jsonl
メモ：s.jsonl は対象セッションファイル。*.jsonl にすればプロジェクト全体を一括集計できます。
