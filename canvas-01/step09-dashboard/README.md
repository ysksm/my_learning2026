# Step 09: 総合演習 — ミニダッシュボードを完成させる

Step01〜08 の全技術を1つのアプリに統合します。
**新しい API は1つも出てきません**。このステップの学びは「組み合わせ方」と
「実アプリの構造」です。

## 1. 完成形の機能

- 無限キャンバス: ドラッグ(何もない場所)でパン、ホイールでマウス中心ズーム(Step04)
- グリッド背景(Step04)、DPR 対応 + ウィンドウリサイズ追従(Step01)
- ウィジェット3種: 時計 / 棒グラフ / カウンタ(Step02, 03)
- クリックで選択、上が勝つヒットテスト(Step05)
- ドラッグで移動(Step06)、ハンドルでリサイズ(Step07)
- 選択枠・ハンドル・ホバー・半透明 HUD(Step08)

## 2. 全体構造 — 実アプリの骨格

```
[状態]
  camera   = { x, y, zoom }
  widgets  = [ { type, x, y, w, h, ... }, ... ]   // 座標はワールド座標
  mode     = 'idle' | 'pan' | 'drag' | 'resize'   // 状態機械(Step07 の拡張)
  selected, hovered, dragOffset, activeHandle, resizeStart

[入力] mousedown / mousemove / mouseup / wheel
  → 状態を書き換えて requestRedraw() するだけ。描画はしない

[描画] draw()  ← requestAnimationFrame でフレームに1回
  1. setTransform(dpr) + clearRect        … Step01, 03
  2. save → camera の変換                  … Step04
  3.   グリッド → ウィジェット → 選択枠/ハンドル … ワールド座標で描く
  4. restore
  5. HUD(スクリーン座標で描く)             … Step08
```

**「入力は状態だけ変える。描画は draw() が状態を見て全部描き直す」**。
この一方通行が Canvas アプリ設計の核心です(React の考え方と同じです)。

## 3. 座標系の交通整理(一番間違えやすい所)

| データ | 座標系 |
|---|---|
| `widgets[i].x, y, w, h` | ワールド |
| マウスイベント (`offsetX` など) | スクリーン |
| ヒットテスト・ドラッグ・リサイズの計算 | **ワールドに変換してから** |
| ハンドルのサイズ、選択枠の線幅 | スクリーン px で一定にしたい → ワールドでは `size / camera.zoom` |

ルールはこれだけです:

```js
function onMouseDown(e) {
  const s = getMousePos(e);              // スクリーン座標
  const p = screenToWorld(s.x, s.y);     // → 最初にワールドへ変換
  // 以降のヒットテストや dragOffset はすべてワールド座標で計算
}
```

「ズームしても見た目のサイズを変えたくないもの」(ハンドル、枠線、グリッド線)は
ワールド座標系の中で `/ camera.zoom` します。demo のコメントで確認してください。

## 4. モード判定の優先順位(mousedown)

```
1. 選択中ウィジェットのハンドル → 'resize'
2. ウィジェット本体            → 選択 + 'drag'(最前面へ)
3. 何もない場所                → 選択解除 + 'pan'
```

Step07 との違いは「3. がパンになる」ことだけです。

## 5. ウィジェットの描画を部品化する(Step03 の translate イディオム)

各ウィジェットの draw 関数は「(0,0) を自分の左上として描く」ように書きます:

```js
function drawWidget(wg) {
  ctx.save();
  ctx.translate(wg.x, wg.y);           // 位置は親(ここ)が面倒を見る
  WIDGET_RENDERERS[wg.type](wg);       // 中身は (0,0)〜(wg.w, wg.h) に描くだけ
  ctx.restore();
}
```

こうすると新しいウィジェット種類を追加するとき、位置やカメラを一切気にせず
「w × h の箱の中に描く関数」を1つ書くだけで済みます。

## 6. アニメーション(時計)と再描画

時計は時刻で表示が変わるので、操作がなくても定期的に再描画が必要です。
demo では単純化のため `requestAnimationFrame` の常時ループにしています
(ダッシュボードは常に何かが動いているため、実用でもこの構成が多い)。
操作時だけ描く省電力設計(Step06 の requestRedraw)とどちらを選ぶかは要件次第です。

## 演習

このステップの演習は穴埋めではなく **機能追加** です。`demo.html` をコピーして
`my-dashboard.html` を作り、以下に挑戦してください(難易度順):

1. **ウィジェット追加**: 「Delete」キーで選択中ウィジェットを削除する
   (ヒント: `window` の `keydown`、`widgets.splice`)
2. **新ウィジェット型**: 折れ線グラフ型 `'line'` を追加する(Step02 demo の流用で OK)
3. **グリッドスナップ**: ドラッグ終了時に位置を 20px 単位に丸める
   (ヒント: `Math.round(x / 20) * 20`)
4. **ダブルクリックで追加**: 空き地をダブルクリックすると新しいウィジェットが生まれる
   (ヒント: `dblclick` イベント + screenToWorld)
5. **永続化**: `localStorage` に widgets と camera を保存し、リロード後も復元する
   (ヒント: `JSON.stringify` / `JSON.parse`。関数は保存できないので type 文字列で持つ設計が活きる)
